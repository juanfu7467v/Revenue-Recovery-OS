# Documentación Técnica del API para el Panel de Control de Revenue-Recovery-OS

Esta documentación detalla los endpoints del API necesarios para el desarrollo de la interfaz de usuario del Panel de Control de Revenue-Recovery-OS. Se enfoca en el flujo de autenticación, la especificación de endpoints por pantalla, el manejo seguro de claves API y la gestión de webhooks, así como un diccionario de códigos de error para una integración robusta.

## 1. Flujo de Autenticación

El sistema utiliza tokens web JSON (JWT) para la autenticación de usuarios, asegurando el acceso seguro a los recursos protegidos del API. El ciclo de vida del JWT es gestionado por el cliente, que debe almacenar y adjuntar el token a las solicitudes subsiguientes.

### 1.1. Registro de Usuario

**Endpoint:** `POST /api/v1/auth/register`

Permite a un nuevo usuario registrarse en el sistema, creando una cuenta de usuario y una organización asociada.

**Cuerpo de la Solicitud (JSON):**

```json
{
  "email": "usuario@ejemplo.com",
  "password": "ContraseñaSegura123",
  "company_name": "Mi Empresa S.A."
}
```

**Respuestas Posibles:**

| Código de Estado | Descripción                                     | Cuerpo de Respuesta (JSON)                                        |
| :--------------- | :---------------------------------------------- | :---------------------------------------------------------------- | 
| `200 OK`         | Usuario y organización creados exitosamente.    | `{"message": "User and organization created successfully", "org_id": "[ID_ORGANIZACION]"}` |
| `400 Bad Request`| El email ya está registrado.                    | `{"detail": "Email already registered"}`                        |
| `500 Internal Server Error`| Error al crear el usuario.                    | `{"detail": "Error creating user."}`                            |
| `503 Service Unavailable`| Conexión a la base de datos no disponible.    | `{"detail": "Database connection not available."}`              |

### 1.2. Inicio de Sesión (Login)

**Endpoint:** `POST /api/v1/auth/login`

Autentica a un usuario existente y devuelve un token de acceso JWT.

**Cuerpo de la Solicitud (Form-data):**

| Campo      | Tipo     | Descripción                                  |
| :--------- | :------- | :------------------------------------------- |
| `username` | `string` | El email del usuario.                        |
| `password` | `string` | La contraseña del usuario.                   |

**Respuestas Posibles:**

| Código de Estado | Descripción                                     | Cuerpo de Respuesta (JSON)                                        |
| :--------------- | :---------------------------------------------- | :---------------------------------------------------------------- | 
| `200 OK`         | Autenticación exitosa.                          | `{"access_token": "[TOKEN_JWT]", "token_type": "bearer"}` |
| `400 Bad Request`| Email o contraseña incorrectos.                 | `{"detail": "Incorrect email or password"}`                     |
| `500 Internal Server Error`| Error durante la autenticación.               | `{"detail": "Error during authentication."}`                    |
| `503 Service Unavailable`| Conexión a la base de datos no disponible.    | `{"detail": "Database connection not available."}`              |

### 1.3. Manejo del Ciclo de Vida del JWT en el Cliente

Una vez obtenido el `access_token`, el cliente debe:

*   **Almacenamiento:** Guardar el token de forma segura (ej. en `localStorage` o `sessionStorage` para aplicaciones web, o `SecureStorage` para móviles).
*   **Uso:** Incluir el token en el encabezado `Authorization` de todas las solicitudes a endpoints protegidos, con el formato `Bearer <TOKEN_JWT>`.
*   **Expiración:** El token tiene una validez de 7 días. El cliente debe estar preparado para manejar respuestas `401 Unauthorized` (ver sección de manejo de errores) y redirigir al usuario a la pantalla de login para obtener un nuevo token.

## 2. Especificación de Endpoints por Pantalla

### 2.1. Dashboard

**Endpoint:** `GET /api/v1/dashboard/metrics`

Obtiene las métricas clave para mostrar en el panel de control del usuario. La autenticación es requerida, y el `org_id` se extrae del token JWT del usuario autenticado.

**Parámetros de Consulta:** Ninguno (la información del usuario se obtiene del token).

**Esquema de Respuesta (JSON):**

```json
{
  "recovered_amount": 1500.75,              // Dinero total recuperado
  "reduction_in_collection_days": 5.2,     // Reducción promedio en días de cobro
  "accounts_at_risk": 10,                  // Cuentas en riesgo de impago
  "monthly_recovered": 1200.50,            // Dinero recuperado este mes
  "recovery_rate": 85.5,                   // Tasa de éxito de recuperación (%)
  "active_recovery_campaigns": 3,          // Campañas de recuperación activas
  "message_summary": "¡Felicidades! Este mes recuperamos $1,200.50 USD que dabas por perdidos"
}
```

### 2.2. Branding

**Endpoint GET:** `GET /api/v1/organization/branding`

Recupera la configuración de branding actual de la organización. Requiere autenticación.

**Esquema de Respuesta (JSON):**

```json
{
  "logo_url": "https://ejemplo.com/mi-logo.png",
  "primary_color": "#3b82f6",
  "secondary_color": "#1e293b",
  "tone": "professional",
  "company_name": "Mi Empresa S.A."
}
```

**Endpoint POST:** `POST /api/v1/organization/branding`

Actualiza la configuración de branding de la organización. Requiere autenticación.

**Cuerpo de la Solicitud (JSON):**

```json
{
  "logo_url": "https://nueva-url.com/logo.png",
  "primary_color": "#FF0000",
  "secondary_color": "#00FF00",
  "tone": "friendly",
  "company_name": "Mi Empresa Renovada"
}
```

**Detalles de los Campos:**

*   `logo_url`: `string` (URL). La URL pública donde se aloja el logo de la empresa. El sistema no gestiona la subida de imágenes directamente, sino que espera una URL accesible.
*   `primary_color`: `string` (código Hex). Color principal de la marca (ej. `#RRGGBB`).
*   `secondary_color`: `string` (código Hex). Color secundario de la marca (ej. `#RRGGBB`).
*   `tone`: `string`. El tono de voz preferido para las comunicaciones. Valores permitidos (ENUM): `professional`, `friendly`, `urgent`.
*   `company_name`: `string`. El nombre de la empresa a mostrar en las comunicaciones.

**Respuestas Posibles (POST):**

| Código de Estado | Descripción                                     | Cuerpo de Respuesta (JSON)                                        |
| :--------------- | :---------------------------------------------- | :---------------------------------------------------------------- | 
| `200 OK`         | Configuración de branding actualizada.          | `{"message": "Branding settings updated successfully"}`         |
| `401 Unauthorized`| Token JWT inválido o ausente.                   | `{"detail": "Could not validate credentials"}`                  |
| `500 Internal Server Error`| Error al actualizar el branding.              | `{"detail": "Failed to update branding: [ERROR_MESSAGE]"}`      |
| `503 Service Unavailable`| Conexión a la base de datos no disponible.    | `{"detail": "Database connection not available."}`              |

### 2.3. Vault (Almacenamiento Seguro de Claves API)

**Endpoint:** `POST /api/v1/organization/vault/api-key`

Permite el envío seguro de claves API de procesadores de pago (como Stripe) para su encriptación y almacenamiento. Requiere autenticación.

**Cuerpo de la Solicitud (JSON):**

```json
{
  "processor": "stripe",
  "api_key": "sk_test_SU_CLAVE_AQUI"
}
```

**Detalles de los Campos:**

*   `processor`: `string`. El nombre del procesador de pago (ej. `stripe`).
*   `api_key`: `string`. La clave API a encriptar y almacenar.

**Seguridad:** La clave API se encripta utilizando el algoritmo Fernet, que proporciona una encriptación simétrica robusta basada en AES-256, garantizando un nivel de seguridad bancario para los datos sensibles [1].

**Respuestas Posibles:**

| Código de Estado | Descripción                                     | Cuerpo de Respuesta (JSON)                                        |
| :--------------- | :---------------------------------------------- | :---------------------------------------------------------------- | 
| `200 OK`         | Clave API almacenada de forma segura.           | `{"message": "API Key for [procesador] stored securely in the vault"}` |
| `400 Bad Request`| Usuario no asociado a una organización.         | `{"detail": "User not associated with an organization"}`        |
| `401 Unauthorized`| Token JWT inválido o ausente.                   | `{"detail": "Could not validate credentials"}`                  |
| `500 Internal Server Error`| Error al almacenar la clave API.              | `{"detail": "Failed to store API Key"}`                         |

### 2.4. Gestión de Webhooks (Historial de Eventos)

**Endpoint:** `POST /api/v1/webhooks/stripe`

Este endpoint recibe los eventos de webhook de Stripe. Aunque es un endpoint de entrada para el sistema, la interfaz de usuario puede consultar los logs generados por este endpoint para mostrar un historial de eventos de recuperación.

**Cuerpo de la Solicitud:** El cuerpo de la solicitud es el payload JSON estándar de un evento de webhook de Stripe. El sistema valida opcionalmente la firma de Stripe si `STRIPE_WEBHOOK_SECRET` está configurado.

**Logs de `recovery_events`:**

Cada evento de webhook recibido se registra en la colección `recovery_events` de Firestore. La interfaz de usuario puede consultar esta colección para mostrar al usuario el historial de webhooks recibidos de Stripe en tiempo real. El esquema de un documento de `recovery_events` es el siguiente:

```json
{
  "event_type": "invoice.payment_failed", // Tipo de evento de Stripe
  "payload": { ... },                     // Payload JSON completo del evento de Stripe
  "received_at": "2023-10-27T10:00:00Z"   // Marca de tiempo de recepción
}
```

**Respuestas Posibles:**

| Código de Estado | Descripción                                     | Cuerpo de Respuesta (JSON)                                        |
| :--------------- | :---------------------------------------------- | :---------------------------------------------------------------- | 
| `200 OK`         | Evento recibido y procesado.                    | `{"status": "success", "message": "Payment failure logged, recovery logic queued"}` (para `invoice.payment_failed`) o `{"status": "success", "message": "Pre-Dunning notification queued"}` (para `customer.subscription.updated`) o `{"status": "success", "message": "Event received"}` (para otros eventos). |
| `400 Bad Request`| Firma de webhook inválida o payload incorrecto. | `{"detail": "[ERROR_MESSAGE]"}`                                 |

# Guía de Integración "Plug & Play" para Clientes de Revenue-Recovery-OS

**Asunto:** Guía de Integración para Clientes de Revenue-Recovery-OS

¡Bienvenido a Revenue-Recovery-OS! Nuestro objetivo es ayudarte a recuperar ingresos perdidos de forma automática y sin complicaciones. Esta guía te mostrará lo fácil que es conectar tu negocio en solo 5 minutos, sin necesidad de ser un experto técnico.

## ¡Integración Súper Fácil en 4 Pasos!

### Paso 1: Conexión Segura con Stripe

Para que Revenue-Recovery-OS pueda trabajar su magia, necesitamos conectar de forma segura con tu cuenta de Stripe. Esto nos permitirá monitorear los pagos fallidos y activar nuestras estrategias de recuperación.

**¿Cómo obtener tu API Key de Stripe?**

1.  Inicia sesión en tu [Dashboard de Stripe](https://dashboard.stripe.com/).
2.  Navega a la sección **Desarrolladores** > **Claves de API**.
3.  Busca la sección **Claves estándar** y copia tu **Clave secreta** (empieza con `sk_live_` o `sk_test_`).

**¡Tu seguridad es nuestra prioridad!**

Una vez que nos proporciones tu API Key, esta se guarda en nuestro **Vault encriptado con nivel bancario (AES-256)**. Esto significa que tu clave está protegida con los más altos estándares de seguridad, asegurando que solo nuestro sistema automatizado pueda acceder a ella de forma segura para realizar las operaciones necesarias. Nunca almacenamos tus claves en texto plano.

### Paso 2: Personalización de Marca (Marca Blanca)

Queremos que tus clientes sientan que las comunicaciones de recuperación provienen directamente de tu negocio, no de un tercero. Por eso, te ofrecemos personalización completa para que los correos de cobro no parezcan spam.

**¿Cómo configurar tu logo y tono de comunicación?**

1.  En tu Panel de Control de Revenue-Recovery-OS, ve a la sección **Branding**.
2.  **Sube tu Logo:** Proporciona la URL pública de tu logo. Este aparecerá en todos los correos electrónicos y comunicaciones de recuperación.
3.  **Elige tus Colores:** Selecciona los colores primario y secundario de tu marca utilizando códigos Hexadecimales (ej. `#FF0000` para rojo). Esto asegurará que las comunicaciones se alineen con la identidad visual de tu empresa.
4.  **Define tu Tono de Voz:** Selecciona el tono que mejor represente a tu marca para las comunicaciones con tus clientes. Puedes elegir entre:
    *   **Profesional:** Un tono formal y directo.
    *   **Cercano:** Un tono amigable y empático.
    *   **Urgente:** Un tono que enfatiza la necesidad de acción inmediata (usar con precaución).

### Paso 3: Activación del Webhook de Stripe

Para que Revenue-Recovery-OS reciba notificaciones en tiempo real sobre pagos fallidos y otras transacciones importantes, necesitas configurar un Webhook en tu cuenta de Stripe. ¡Es más fácil de lo que suena!

**Instrucciones paso a paso:**

1.  En tu Panel de Control de Revenue-Recovery-OS, encontrarás una **URL de Webhook** única para tu cuenta. Cópiala.
2.  Inicia sesión en tu [Dashboard de Stripe](https://dashboard.stripe.com/).
3.  Navega a **Desarrolladores** > **Webhooks**.
4.  Haz clic en **Agregar un endpoint**.
5.  Pega la URL de Webhook que copiaste de Revenue-Recovery-OS en el campo **URL del endpoint**.
6.  En **Eventos a enviar**, selecciona los siguientes eventos (o todos los eventos si no estás seguro):
    *   `invoice.payment_failed`
    *   `customer.subscription.updated`
7.  Haz clic en **Agregar endpoint**.

¡Listo! Ahora Stripe nos enviará automáticamente la información necesaria para iniciar el proceso de recuperación.

### Paso 4: Prevención Proactiva (Pre-Dunning)

Revenue-Recovery-OS no solo recupera pagos fallidos, sino que también ayuda a prevenirlos. Con nuestra función de Pre-Dunning, te avisaremos a ti y a tus clientes antes de que su tarjeta de crédito o débito esté a punto de vencer.

**¿Cómo funciona?**

Nuestro sistema monitorea las fechas de vencimiento de las tarjetas de tus suscriptores. Días antes de que una tarjeta expire, Revenue-Recovery-OS puede enviar un recordatorio amigable a tu cliente, invitándolo a actualizar su información de pago. Esto evita que el pago llegue a fallar en primer lugar, reduciendo la fricción y asegurando la continuidad de tus ingresos.
